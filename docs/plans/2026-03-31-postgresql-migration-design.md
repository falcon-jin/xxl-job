# PostgreSQL Dual-Database Migration Design

## Goal

Add PostgreSQL support to `xxl-job-admin` while preserving existing MySQL support. The system must run against either database at deployment time, select the correct SQL dialect at runtime, and provide automated PostgreSQL regression coverage for the SQL statements that currently depend on MySQL-specific behavior.

## Current Context

- `xxl-job-admin` uses Spring Boot, HikariCP, and MyBatis XML mappers.
- The repository currently ships only MySQL defaults:
  - MySQL JDBC driver in the root dependency management and `xxl-job-admin/pom.xml`
  - MySQL datasource example in `xxl-job-admin/src/main/resources/application.properties`
  - MySQL bootstrap in `docker-compose.yml`
  - MySQL DDL in `doc/db/tables_xxl_job.sql`
- SQL dialect differences are concentrated in MyBatis XML files under `xxl-job-admin/src/main/resources/mapper`.

## Observed MySQL-Specific Dialect Points

- DDL:
  - `AUTO_INCREMENT`
  - `ENGINE = InnoDB`
  - `DEFAULT CHARSET = utf8mb4`
  - `CREATE database`, `use`, `SET NAMES`
- DML and queries:
  - `LIMIT #{offset}, #{pagesize}`
  - `IFNULL(...)`
  - `DATE_ADD(..., INTERVAL -#{timeout} SECOND)`
  - `ON DUPLICATE KEY UPDATE`
  - backtick-quoted identifiers
- Affected mapper files:
  - `XxlJobGroupMapper.xml`
  - `XxlJobInfoMapper.xml`
  - `XxlJobLogMapper.xml`
  - `XxlJobLogGlueMapper.xml`
  - `XxlJobLogReportMapper.xml`
  - `XxlJobRegistryMapper.xml`
  - `XxlJobUserMapper.xml`

## Chosen Architecture

Use MyBatis `databaseId` to switch SQL dialects at runtime.

- Keep one set of Java mapper interfaces and service/controller code.
- Keep common SQL statements shared when syntax is already cross-database.
- Add dialect-specific SQL variants only for statements that differ between MySQL and PostgreSQL.
- Resolve the active dialect through a MyBatis `DatabaseIdProvider` bean.

This keeps the change localized to configuration, mapper XML, DDL, and tests. It avoids pushing dialect branching into Java services and avoids a broader persistence rewrite.

## Database Compatibility Model

The supported deployment modes are:

- MySQL
  - Existing behavior remains the default and must not regress.
- PostgreSQL
  - New DDL, datasource examples, docker wiring, and regression tests are added.

Only one database is active per deployment. The codebase supports both.

## DDL Strategy

Keep the existing MySQL script and add a PostgreSQL script:

- Existing:
  - `doc/db/tables_xxl_job.sql`
- New:
  - `doc/db/tables_xxl_job_postgresql.sql`

PostgreSQL DDL rules:

- Remove MySQL-only bootstrap directives:
  - `CREATE database`
  - `use`
  - `SET NAMES`
  - `ENGINE`
  - `DEFAULT CHARSET`
- Convert identity columns:
  - `AUTO_INCREMENT` -> `GENERATED BY DEFAULT AS IDENTITY`
- Convert types:
  - `int(11)` -> `integer`
  - `bigint(20)` -> `bigint`
  - `tinyint(4)` -> `smallint`
  - `datetime` -> `timestamp`
  - `mediumtext` -> `text`
- Keep timestamps as `timestamp without time zone` to match current application semantics and avoid introducing implicit time zone conversions.
- Replace MySQL index syntax with PostgreSQL-native `UNIQUE` constraints or unique indexes.
- Rewrite seed data so multi-line text values are safe in PostgreSQL, using dollar-quoted literals where needed.

## SQL Dialect Strategy

### Shared SQL

Statements that are already portable should remain unqualified and shared across both databases.

### Dialect-Specific SQL

Statements with syntax differences should be duplicated under the same mapper `id` with explicit `databaseId` values:

- `databaseId="mysql"`
- `databaseId="postgresql"`

Expected conversions:

- Pagination
  - MySQL: `LIMIT #{offset}, #{pagesize}`
  - PostgreSQL: `LIMIT #{pagesize} OFFSET #{offset}`
- Null handling
  - MySQL: `IFNULL(x, 0)`
  - PostgreSQL: `COALESCE(x, 0)`
- Time arithmetic
  - MySQL: `DATE_ADD(ts, INTERVAL -#{timeout} SECOND)`
  - PostgreSQL: `ts - (#{timeout} * INTERVAL '1 second')`
- Upsert
  - MySQL: `ON DUPLICATE KEY UPDATE`
  - PostgreSQL: `ON CONFLICT (...) DO UPDATE`
- Identifier quoting
  - Remove backticks from portable statements and PostgreSQL variants

## Mapper Conversion Scope

Priority files and behaviors:

- `XxlJobRegistryMapper.xml`
  - dead registry lookup
  - alive registry lookup
  - registry upsert
- `XxlJobLogReportMapper.xml`
  - report upsert
- `XxlJobLogMapper.xml`
  - pagination
  - aggregate metrics
  - cleanup subqueries
  - failed log lookup
- `XxlJobGroupMapper.xml`
  - pagination
- `XxlJobInfoMapper.xml`
  - pagination
  - scheduled job fetch
- `XxlJobUserMapper.xml`
  - pagination
- `XxlJobLogGlueMapper.xml`
  - cleanup subquery with limit

## Generated Key Expectations

Insert statements currently rely on `useGeneratedKeys="true"` and `keyProperty="id"`.

The migration must verify that generated key backfill still works for PostgreSQL on:

- `xxl_job_group`
- `xxl_job_info`
- `xxl_job_log`
- `xxl_job_logglue`
- `xxl_job_log_report`
- `xxl_job_user`
- `xxl_job_registry`

If MyBatis or the PostgreSQL driver fails to backfill keys in a specific mapper, add the smallest necessary dialect-specific insert strategy rather than changing all inserts preemptively.

## Transaction and Locking

`XxlJobLockMapper.xml` uses `SELECT ... FOR UPDATE`. PostgreSQL supports this, but the migration must verify the lock behavior inside a transaction-backed integration test because scheduler correctness depends on it.

## Configuration Changes

The codebase needs:

- PostgreSQL JDBC driver dependency
- MyBatis `DatabaseIdProvider` bean
- PostgreSQL datasource examples for local and docker usage
- Optional test-only datasource overrides for PostgreSQL integration tests

The default application properties may stay MySQL-oriented, but the documentation and docker setup must clearly show both supported database choices.

## Testing Strategy

Add PostgreSQL integration coverage with Testcontainers.

Goals:

- No manual local PostgreSQL requirement for contributors
- Schema initialized from the PostgreSQL DDL script
- Spring Boot context boots against PostgreSQL
- Dialect-sensitive mapper behaviors are validated automatically

Minimum PostgreSQL regression set:

- `XxlJobRegistryMapper`
  - `findDead`
  - `findAll`
  - `registrySaveOrUpdate`
- `XxlJobLogReportMapper`
  - `saveOrUpdate`
- `XxlJobLogMapper`
  - pagination
  - aggregate metrics
  - cleanup query
  - generated key backfill
- `XxlJobGroupMapper`
  - generated key backfill
  - pagination
- `XxlJobInfoMapper`
  - generated key backfill
  - pagination
- `XxlJobLockMapper`
  - `FOR UPDATE` transaction behavior

MySQL remains the default runtime path. The first implementation phase does not need a full dual-database CI matrix for every existing test class, but it must add targeted PostgreSQL coverage for the mapper methods that have dialect-specific behavior.

## Acceptance Criteria

The migration is complete when all of the following are true:

- MySQL startup and existing default configuration still work.
- PostgreSQL can initialize from `doc/db/tables_xxl_job_postgresql.sql`.
- `xxl-job-admin` starts against PostgreSQL.
- MyBatis selects the correct SQL per active database.
- PostgreSQL regression tests pass for upsert, time arithmetic, pagination, generated keys, and row locking.
- User-facing docs and docker examples show both MySQL and PostgreSQL usage.

## Risks

- PostgreSQL generated key behavior may differ per mapper.
- Subqueries with pagination in cleanup SQL are easy to regress during dialect conversion.
- Upsert behavior must preserve existing uniqueness assumptions exactly.
- Time arithmetic must remain semantically equivalent for registry heartbeat expiration.
- Documentation drift is likely unless README, official docs, and docker examples are updated in the same change.

## Non-Goals

- Rewriting persistence from MyBatis XML to another ORM or query builder
- Adding simultaneous multi-database operation inside one running admin instance
- Broad schema redesign unrelated to database compatibility
